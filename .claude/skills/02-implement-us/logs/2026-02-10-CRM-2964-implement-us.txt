- Related Epic: CRM-2610 (Kündigung über Events flach auf dem Case abbilden)

## Requirements
The VO (Vertriebsorganisation) needs to add a remark per termination event to
document performed actions (e.g., "Kündigungsbestätigung verschickt und in ELO
abgelegt"). Each event should have its individual remark, while the Case
maintains a full history of all remarks in descending chronological order.

## Acceptance Criteria
- [x] New rich text field STLGS_NewRemarks__c on Case for VO to enter remarks
- [x] Field visible on EditCase FlexiPage when Type = Kündigung
- [x] On save, remark is stored in the created Event's GeneralDescription
- [x] On save, remark is prepended (with timestamp + username) to Case.STLGS_GeneralDescription__c
- [x] Case.STLGS_GeneralDescription__c shows all remarks in descending order
- [x] Event.STLGS_GeneralDescription__c shows only the single remark for that event
- [x] VO user has overview on case level and detailed view per event

## Implementation Approach
**Mixed: Declarative + Apex**

- **Declarative**: New custom field, flow modification, FlexiPage update, permission sets
- **Programmatic**: Apex Trigger Action for rich text concatenation with timestamps
  (Flow cannot reliably handle rich text HTML concatenation with dynamic timestamps)

### Rationale
- The timestamp + username header formatting and HTML concatenation logic is best
  handled in Apex (BeforeUpdate trigger action) for reliability and testability
- The existing flow STLGS_CreateUpdateTerminationEvent handles event creation and
  only needs a field mapping change (declarative)
- The Trigger Action Framework is already in use for Case, making this the natural pattern

## Key Design Decisions

1. **BeforeUpdate Trigger Action** (not AfterUpdate or Flow):
   - Runs before the record is saved, so when the AfterSave flow fires,
     both STLGS_NewRemarks__c (for the event) and STLGS_GeneralDescription__c
     (accumulated history) are already set correctly
   - No additional DML needed, no recursion risk

2. **Descending order with HTML headers**:
   Format: <p><strong>DD.MM.YYYY HH:mm - Username</strong></p>{remark}<hr/>{existing}
   - Newest remarks appear first for easy scanning
   - Timestamp + username provides audit trail
   - <hr/> separates entries visually

3. **Field NOT auto-cleared after save**:
   - The trigger action only fires when STLGS_NewRemarks__c CHANGES
   - If the user re-saves without editing the remark, no duplicate entry is created
   - User naturally overwrites with the next remark

4. **GeneralDescription made readonly on FlexiPage**:
   - Prevents manual edits to the accumulated history
   - Users enter remarks only via STLGS_NewRemarks__c

5. **Flow change is minimal**:
   - Only 2 lines changed in the flow XML (field references)
   - Avoids restructuring the complex flow logic
