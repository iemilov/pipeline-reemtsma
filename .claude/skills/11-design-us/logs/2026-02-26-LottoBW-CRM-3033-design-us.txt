Summary: Der bestehende Screen Flow für die Erstellung von Rücklastschrift-Sub-Cases im Kontoauszug
verwendet eine flowruntime:lookup-Komponente, die den Case.ContactId Lookup-Filter nicht im
Flow-Kontext auswerten kann ($Source.RecordTypeId nicht verfügbar).

== Step 2: Explore Codebase ==

1. Identified correct flow: STLG_New2ndLevelCase (NOT STLGS_CreateReturnDebit)
   - B2C flow for creating 2nd Level sub-cases (Rücklastschrift, Offener Betrag, Zahlungsanweisung)
   - Launched from parent Kontoauszug case on STLG_SecondLevelCaseRecordPage
   - recordId = parent Case ID
   - Sets ParentId = recordId on new case

2. Problematic component: STLG_Account (flowruntime:lookup)
   - fieldApiName = ContactId, objectApiName = Case
   - Despite "Account" label, this is a Contact lookup
   - STLG_Case.ContactId = STLG_Account.recordId (line 167-172)

3. Case.ContactId lookup filter (ContactId.field-meta.xml):
   - Status: active = FALSE
   - Logic: (1 AND 2) OR (3 AND 4)
   - B2C (Erwin/Standard/SecondLevel): IsPersonAccount = true
   - B2B (Edit/Control/Sales/ServiceDesk/TestPurchase): RecordType = SalesContact
   - Filter uses $Source.RecordTypeId which cannot be resolved in flow context

4. FlexiPage: STLG_SecondLevelCaseRecordPage
   - Flow embedded directly (not Quick Action)
   - Visibility: TopicDetail = "Kontoauszug/Rücklastschriften" AND Parent.Subject is empty
   - Passes recordId = {!Record.Id}

5. Existing LWC patterns reviewed:
   - stlg_createCase: Case creation via lightning-record-edit-form
   - stlgs_editCase: Case status update via updateRecord
   - stlg_searchSubCasesList: GraphQL query for sub-cases list

6. Domain docs: pipeline/customers/lotto-bw/docs/ruecklastschrift.md reviewed

== Step 3: Resolve Open Questions ==

Q1: Naming prefix? (Jira = B2C, existing metadata = STLGS_)
A1: STLG_ (B2C) — per user

Q2: Account field behavior?
A2: Filtered lookup

Q3: ParentId for new case?
A3: Yes — Kontoauszug case is always parent of Rücklastschrift cases

Q4: What happens to existing flow?
A4: Keep flow, embed LWC as screen component into the flow (not replace the flow)

Q5: Entry point — where is the flow launched from?
A5: Clarification: This is about B2C 2nd Level Cases, NOT the B2B Rücklastschrift flow.
    The flow is already embedded in the SecondLevelCaseRecordPage.

Q6: Contact filter logic?
A6: Use existing Case.ContactId lookup filter logic (IsPersonAccount = true for B2C)

== Step 4: Implementation Approach ==

Solution: Create LWC flow screen component (stlg_contactLookupFilter) that:
- Uses lightning-record-picker with filter: IsPersonAccount = true
- Outputs selectedContactId and selectedAccountId to flow
- Replaces flowruntime:lookup in STLG_New2ndLevelCase flow screen
- No Apex required — standard wire adapters only
