## Ausgangslage

User meldet Bug bei Account 210460 Banzhaf:
- TK 04.02.2026 (Ersttest Jugendschutz): nicht bestanden → Ampelstatus "TK Schließung" → Kündigung korrekt erstellt
- TK 07.02.2026 (Nachtest Spielerschutz): bestanden → Ampelstatus bleibt "TK Schließung" → Kündigung fälschlicherweise erstellt

## Analyse

### Beteiligte Komponenten

1. Flow: STLGS_CalculateTestPurchaseStatus.flow-meta.xml
   - Trigger: Case Status changed to "Testkaufergebnis geprüft" (RecordAfterSave)
   - Immediate Path: Ruft Apex Invocable zur Ampelstatus-Berechnung
   - Async Path: Prüft ob Ampelstatus "geschlossen" → erstellt Kündigung

2. Apex: STLGS_CalculateTestPurchaseStatus.cls
   - Invocable Action für Ampelstatus-Berechnung
   - Nutzt Custom Metadata Mappings für Status-Übergänge
   - Berechnet Penalties

3. Custom Metadata: STLGS_TestPurchaseStatusChanges.geschlossen_all.md-meta.xml
   - InputStatus: geschlossen
   - OutputStatusPositive: geschlossen (bestanden → bleibt geschlossen)
   - OutputStatusNegative: geschlossen (nicht bestanden → bleibt geschlossen)

### Root Cause

Flow Decision "STLGS_Geschlossen" (Zeile 23-44) prüft nur:
  $Record.STLGS_TestPurchaseStatus__c == "geschlossen"

Es fehlt die Prüfung:
  $Record.STLGS_ResultTestPurchase__c == "nicht bestanden"

Dadurch wird auch bei bestandenem TK ein Kündigungsvorgang erstellt, wenn der Ampelstatus
bereits "geschlossen" war (Output bleibt "geschlossen" laut CMT).
