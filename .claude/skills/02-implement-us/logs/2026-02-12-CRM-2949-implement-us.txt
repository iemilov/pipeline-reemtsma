STORY DETAILS
================================================================================
Title: Kündigungen v2 - Status Reset bei Änderung durch RD
Epic: CRM-2610 (Kündigungsprozess)

Requirements:
1. Automatic Status Reset on Changes by RD
   - When the RD (Regionaldirektor) makes ANY adjustment to a termination case
     (Kündigungsvorgang), the Case should automatically be reset to the status
     "Übergeben an VO" (Handed over to VO)
   - Goal: The VO (Vertriebsorganisation) is actively informed about changes
     and can react accordingly (e.g., send a new confirmation letter)
2. Owner reassignment to STLGS_SalesOrganization queue
3. Increment STLGS_CounterHandoverVO__c counter

================================================================================
CODEBASE ANALYSIS
================================================================================
Key Findings:
1. RD Identification:
   - RD users identified by Profile.Name = "STLGS_Sales"
   - Pattern established in STLGS_AdaptTermination.flow-meta.xml (line 540)
   - Also used in stlgs_handoverCase.js LWC

2. Termination Cases:
   - Identified by: STLGS_Type__c = "Kündigung"
   - Record Types: STLGS_EditCase, STLGS_ReadOnlyCase
   - Business Process: STLGS_EditProcess

3. Existing Infrastructure:
   - STLGS_CounterHandoverVO__c field (Number, Precision 3, Default 0)
   - STLG_UpdateCaseOnQueue.flow-meta.xml (Before Save, Order 1300)
   - STLGS_SalesOrganization queue for VO team

4. Validation Rules to consider:
   - STLGS_TerminationMandatoryFields: Requires STLGS_IsTerminationElo__c = true
   - STLGS_ValidateTerminationLetterDate: Requires STLGS_TerminationLetterDate__c filled
   Decision: Option B - Flow entry conditions check these fields upfront

================================================================================
IMPLEMENTATION APPROACH
================================================================================
Chosen: Record-Triggered Flow (Before Save)

Rationale:
- Follows project convention: prefer declarative solutions over code
- Consistent with existing patterns (STLG_UpdateCaseOnQueue uses same approach)
- No complex business logic requiring Apex
- Before Save → no additional DML needed

Alternative Considered: Trigger Action (Apex)
- Rejected: Unnecessary code complexity for simple status change logic

================================================================================
IMPLEMENTATION DETAILS
================================================================================
Flow: STLGS_ResetTerminationOnRDChange
Type: Record-Triggered Flow (Before Save)
Object: Case
Trigger: Update
API Version: 64.0
Status: Active
Trigger Order: 1000 (before STLG_UpdateCaseOnQueue at 1300)

Entry Conditions (all AND):
1. STLGS_Type__c = "Kündigung"
2. Status != "Übergeben an VO" (prevent no-op)
3. Status != "Closed"
4. Status != "Cancelled"
5. STLGS_IsTerminationElo__c = true (validation rule guard)
6. STLGS_TerminationLetterDate__c IS NOT NULL (validation rule guard)

Flow Logic:
1. START: Case update triggers flow with entry conditions
2. Get Running User: Query User WHERE Id = $User.Id → get Profile.Name
3. Decision: Is User RD?
   - Condition: STLGS_GetRunningUser.Profile.Name = "STLGS_Sales"
   - If Yes → Get VO Queue
   - If No → End (no action)
4. Get VO Queue: Query Group WHERE DeveloperName = "STLGS_SalesOrganization"
   AND Type = "Queue"
5. Assignment (Reset Status and Owner):
   - $Record.Status = "Übergeben an VO"
   - $Record.OwnerId = STLGS_GetVOQueue.Id
   - $Record.STLGS_CounterHandoverVO__c = IncrementedHandoverCounter

Formula:
- IncrementedHandoverCounter:
  IF(ISBLANK({!$Record.STLGS_CounterHandoverVO__c}), 1,
     {!$Record.STLGS_CounterHandoverVO__c} + 1)
