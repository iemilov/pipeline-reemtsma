## Jira Story
- Key: CRM-2964
- Summary: Kündigungen v2 - Bemerkung pro Event für VO
- Component: B2B
- Status: Next

## Requirements
Die VO (Vertriebsorganisation) soll pro Kündigungs-Event eine Bemerkung erfassen
können, um durchgeführte Aktionen zu dokumentieren.

Beispiele für Bemerkungen:
- Kündigungsbestätigung verschickt und in ELO abgelegt
- Bestätigung Verschiebung verschickt und abgelegt
- Rücknahmebestätigung versendet

Additional requirements:
- Sales persons type a new remark via a staging field on the Case
- This creates a new termination event with ONLY the new text
- After event creation, ALL event remarks are aggregated back to
  Case.STLGS_GeneralDescription__c as a cumulative history

## Implementation Approach: Declarative (Fields + Flows + Layout + Permissions)

### Architecture
Two flows work together in a cascade pattern:

**Flow A: STLGS_CreateRemarkEventFromCase (on Case After Save)**
- Entry: STLGS_NewRemarkVO__c is changed AND not blank
- Creates STLGS_Event__c with RemarkVO = the entered text
- Clears STLGS_NewRemarkVO__c on Case
- No recursion: clearing the field makes the entry condition fail

**Flow B: STLGS_RollupEventRemarksToCase (on STLGS_Event__c After Save)**
- Triggered by the event creation from Flow A (cascade)
- Queries all events for the parent Case with non-null RemarkVO
- Concatenates all remarks as HTML: <p><b>DD.MM.YYYY - Type:</b> Remark</p>
- Updates Case.STLGS_GeneralDescription__c with aggregated content
- Also fires when someone edits an event's remark directly

### Data Flow
1. Sales person types remark in Case.STLGS_NewRemarkVO__c and saves
2. Flow A creates STLGS_Event__c with STLGS_RemarkVO__c = the new text
3. Flow A clears STLGS_NewRemarkVO__c on Case
4. Event creation triggers Flow B (cascade)
5. Flow B aggregates ALL event remarks → Case.STLGS_GeneralDescription__c
6. Case.STLGS_GeneralDescription__c now shows full history (read-only on layout)

## Post-Deployment Steps
1. Verify STLGS_Event__c object exists in the target org
2. Assign STLGS_Event__c-Event Layout to STLGS_TerminationRecordType
3. Verify both flows are active:
   - STLGS_CreateRemarkEventFromCase
   - STLGS_RollupEventRemarksToCase
4. Test the workflow:
   a. Open a termination Case (STLGS type with TerminationType set)
   b. Type a remark in "Neue Bemerkung VO"
   c. Save the Case
   d. Verify a new STLGS_Event__c was created with the remark
   e. Verify Case.STLGS_GeneralDescription__c shows the aggregated history
   f. Verify "Neue Bemerkung VO" was cleared
   g. Add another remark and verify both appear in the history

## Assumptions
- STLGS_Event__c custom object exists in the target org
- Case.STLGS_GeneralDescription__c is used exclusively for aggregated
  event remarks on termination cases (flow overwrites the field)
- Case.STLGS_TerminationType__c is set before remarks are entered
  (used as the event type on created events)
- Event deletion does not trigger re-aggregation (only Create/Update)
