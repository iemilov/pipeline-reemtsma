## Kritische Befunde (HIGH)

### K1: Fehlende Autorisierung auf User-Management-Endpoints

- **Dateien:** `functions/api/users.js`, `functions/api/users/[id].js`
- **Problem:** `GET/POST /api/users` und `PATCH/DELETE /api/users/:id` pruefen nur die JWT-Authentifizierung, aber nicht ob der Benutzer Admin-Rechte hat. Jeder authentifizierte User kann:
  - Alle Admin-Accounts auflisten
  - Neue Admin-Accounts erstellen
  - Passwoerter anderer Admins aendern
  - Admins deaktivieren oder loeschen
- **Auswirkung:** Privilege Escalation â€” ein normaler User koennte sich selbst Admin-Rechte geben
- **Fix:** Rolle/Berechtigung in JWT-Payload aufnehmen und in den Endpoints pruefen. Da aktuell alle User gleichzeitig Admins sind, alternativ ein `is_superadmin`-Flag einfuehren.

### K2: Interne Fehlermeldungen an Clients exponiert

- **Dateien:** `functions/api/cards/sync.js:14`, `functions/api/cards/[id]/prompt.js:116`, `functions/api/users.js:25`, `functions/api/releases.js:20`
- **Problem:** Exception-Messages werden direkt an den Client zurueckgegeben (`{ error: err.message }`). Fehlermeldungen koennen interne Pfade, DB-Details oder API-Response-Details enthalten.
- **Auswirkung:** Information Disclosure â€” Angreifer koennen interne Infrastruktur-Details erfahren
- **Fix:** Generische Fehlermeldungen an Clients (`"Internal server error"`) zurueckgeben, Details nur serverseitig loggen.

## Wichtige Befunde (MEDIUM)

### W1: Rate Limiting nur auf Login-Endpoint

- **Datei:** `functions/api/_middleware.js:3-18`
- **Problem:** Rate Limiting ist nur auf `/api/auth/login` angewendet. Teure Operationen sind ungeschuetzt:
  - `/api/cards/sync` â€” loest kompletten GitHub-Sync aus
  - `/api/cards/:id/prompt` â€” Claude-API-Aufrufe (kostenintensiv)
  - `/api/skills/run` â€” startet GitHub-Actions-Workflows
- **Fix:** Per-User Rate Limiting fuer authentifizierte Endpoints einfuehren, mindestens fuer AI- und Sync-Endpoints.

### W2: Race Condition im Rate Limiting

- **Datei:** `functions/api/_middleware.js:6-18`
- **Problem:** Rate-Limit-Check liest den Zaehler und aktualisiert ihn in separaten Queries. Bei gleichzeitigen Requests koennen mehrere den gleichen Zaehlerstand lesen und alle den Limit umgehen.
- **Fix:** Atomare Inkrement-Operation verwenden (`UPDATE ... SET count = count + 1 RETURNING count`).

### W3: Keine Error Boundary im Frontend

- **Dateien:** `src/main.jsx`, `src/App.jsx`
- **Problem:** Die App hat keine Top-Level Error Boundary. Ein unkontrollierter Fehler in einer Komponente crasht die gesamte Anwendung.
- **Fix:** `<ErrorBoundary>` Wrapper um `<App />` in `main.jsx` hinzufuegen mit Fallback-UI.

### W4: Memory Leaks in CardModal

- **Datei:** `src/components/CardModal.jsx:197-217`
- **Problem:** Skill-Polling-Intervall wird in einem Event-Handler gestartet, nicht in einem Effect. Wenn das Modal schliesst waehrend Polling aktiv ist, laeuft das Intervall im Hintergrund weiter.
- **Fix:** Polling in `useEffect` mit Cleanup-Return verlagern. Timeouts in Refs speichern und bei Unmount clearen.

### W5: Memory Leak in useBoard

- **Datei:** `src/hooks/useBoard.js:122-134`
- **Problem:** Der `persistOrder` Debounce-Timeout wird nicht bei Unmount bereinigt. API-Calls koennten nach Navigation noch feuern.
- **Fix:** Cleanup-Effect hinzufuegen der `clearTimeout(reorderTimeout.current)` ausfuehrt.

### W6: TOCTOU-Race-Condition bei Card Reorder

- **Datei:** `functions/api/cards/reorder.js:11-23`
- **Problem:** Spalten-Zuweisungen werden gelesen (Zeile 17-19), dann separat aktualisiert (Zeile 23). Parallele Requests koennten inkonsistente Zustaende erzeugen.
- **Fix:** Lese- und Schreiboperation in einem D1 Batch ausfuehren.

### W7: Keine Transaktionskontrolle im Sync

- **Datei:** `functions/lib/sync.js:47-51`
- **Problem:** Sync-Operationen laufen ohne explizite Transaktion. Bei Unterbrechung koennte die DB in einem inkonsistenten Zustand sein (einige Cards aktualisiert, andere nicht).
- **Fix:** `db.batch()` fuer zusammengehoerende Operationen verwenden.

### W8: CardModal zu gross (475+ Zeilen)

- **Datei:** `src/components/CardModal.jsx`
- **Problem:** Eine Komponente verwaltet Form-State (12 State-Variablen), AI-Features, Skill-Ausfuehrung, Validierung und Rendering.
- **Fix:** Aufteilen in: `DescriptionEditor`, `SubtaskManager`, `TagPicker`, `SkillControls`

### W9: Stille Fehler bei API-Aufrufen

- **Dateien:** Mehrere Stellen in `CardModal.jsx:143-164`, `CommandPalette.jsx:44-47`, `useBoard.js:79`
- **Problem:** API-Fehler werden mit `catch { /* silently fail */ }` oder `.catch(() => {})` verschluckt. Benutzer erfahren nicht, dass eine Aktion fehlgeschlagen ist.
- **Fix:** Konsistente Fehlerbehandlungsstrategie implementieren (Toast-Notifications oder Error-State).

### W10: Stille Fehler im Backend

- **Dateien:** `functions/api/skills/run.js:137,145,157`, `functions/api/skills/runs.js:9`
- **Problem:** DB-Updates in Background-Tasks werden mit `.catch(() => {})` ignoriert. Fehler werden nicht einmal geloggt.
- **Fix:** Mindestens `console.error` in Catch-Bloecke einfuegen.

### W11: Deprecated `unescape()` Funktion

- **Datei:** `functions/lib/github-md.js:181`
- **Problem:** `btoa(unescape(encodeURIComponent(content)))` verwendet die deprecated Funktion `unescape()`.
- **Fix:** Ersetzen mit `btoa(String.fromCharCode(...new TextEncoder().encode(content)))` oder einer expliziten UTF-8-Encoding-Hilfsfunktion.

### W12: Verwaiste Webhook-Route in Middleware

- **Datei:** `functions/api/_middleware.js:41-44`
- **Problem:** Middleware erlaubt `/api/github/webhook` ohne Auth-Check, aber der Webhook-Handler wurde geloescht. Tote Code-Referenz.
- **Fix:** Webhook-Bypass aus der Middleware entfernen.

## Architektur-Bewertung

### Staerken

- **Edge-First-Architektur:** Cloudflare Pages + D1 bietet exzellente Performance und globale Verfuegbarkeit
- **Bidirektionaler MD-Sync:** Elegantes Design mit GitHub als Source of Truth und D1 als schneller Cache
- **Optimistische Updates:** Sofortige UI-Reaktion mit asynchroner Server-Synchronisation
- **Saubere API-Struktur:** Konsistente REST-Konventionen mit klarer Trennung
- **Cookie-Sicherheit:** HttpOnly, Secure, SameSite=Strict â€” vorbildlich
- **SQL-Injection-Schutz:** Alle Queries parameterisiert
- **Keine hardcoded Secrets:** Alle sensiblen Werte aus Environment

### Verbesserungspotenzial

- **Autorisierung:** Aktuell gibt es nur Authentifizierung (ist eingeloggt?), aber keine Autorisierung (darf das tun?)
- **Error Handling:** Inkonsistente Strategie â€” Mix aus stillen Fehlern, Console-Logs und Alert-Dialogen
- **Komponenten-Groesse:** CardModal mit 475+ Zeilen ist zu gross und schwer wartbar
- **Memory Management:** Mehrere potenzielle Memory Leaks durch fehlende Cleanup-Funktionen
- **Monitoring:** Keine strukturierte Fehlerprotokollierung oder Alerting

### Skalierbarkeit

- D1 als SQLite-basierte DB ist fuer moderate Last gut geeignet
- GitHub-API-Calls beim Board-Load koennen bei vielen Cards zum Bottleneck werden (sequential file reads)
- Kein Caching-Layer zwischen D1 und GitHub â€” jeder Board-Load triggert vollen Sync
